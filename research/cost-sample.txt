In this conversation we set out to add voice input to the Grok Build extension for Visual Studio Code. The first question was whether it was even possible. I probed the Grok command line interface and found that it advertises prompt capabilities with audio set to false, and when I sent an audio content block over the Agent Client Protocol, the binary rejected it with an invalid parameters error, calling it an unsupported content block. That told us the coding agent itself cannot ingest audio. But the platform can: xAI ships a separate speech to text product, billed pay as you go on a console developer key, completely independent from the command line login and from any subscription. Because Visual Studio Code webviews cannot reach the microphone, we capture audio in the extension host using an ffmpeg child process, the same place the command line and terminals are spawned, and then we post the recorded clip straight to the speech to text service. We verified the whole pipeline end to end on native Windows: device detection parsed the real microphone name from ffmpeg, a live three second capture produced a finalized wave file, and the shipped transcription function returned the exact spoken sentence. We added a microphone button in the top right corner of the message box that turns blue with animated waves while listening. When no key is configured, the button shows a small setup dot and a tooltip, and clicking it offers an actionable open settings or get a key prompt. We also added a hands free send command: ending a dictation with the phrase grok send strips the phrase and submits the message automatically, and the two word phrase is deliberate so it does not fire on a message that merely ends in the word send. Then we moved to streaming. The batch flow required a click to start and a click to stop before any transcription happened, which meant the spoken send command could not work without a manual click first. The streaming endpoint solves this: it is a websocket that accepts raw pulse code modulation audio frames and returns transcript events as you speak, with interim partial results and smart turn end of turn detection. We confirmed the protocol empirically, discovering that the transcript actually lives in the partial events keyed by their start time, because the final done event can arrive empty when smart turn finalizes the utterance mid stream. We also confirmed that the key term query parameter biases the model toward recognizing specific phrases, which directly addresses the mishearing where grok send came back as a single garbled word. Throughout, every pure helper was unit tested, the live calls stayed in manual research probes, and the test count grew while remaining completely free of the Grok binary in continuous integration. This sample paragraph is deliberately built from the real words of our working session so that the transcription cost we measure reflects genuine dictation rather than an artificial benchmark, and the numbers we report can be trusted as representative of everyday use.
